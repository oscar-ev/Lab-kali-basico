FASE 1 — RECONHECIMENTO COM NMAP
01

Descobrir portas e serviços do alvo

O nmap é o scanner de rede mais usado no mundo. Vamos usá-lo para mapear o que está rodando no alvo.

nmapport scanservice detection
⏱ 15 min
🎯 O QUE VOCÊ VAI DESCOBRIR Quais portas estão abertas no alvo, quais serviços rodam em cada porta e qual versão do software está sendo usada.
1.1
Instalar o nmap no Kali
KALI — root@kali
root@kali#apt update -qq && apt install -y nmap nikto curl
Atualiza a lista de pacotes e instala as 3 ferramentas da aula de uma vez. Aguarde terminar (leva ~2 min).
1.2
Varredura básica — quais portas estão abertas?
KALI — root@kali
root@kali#nmap localhost
nmap localhost varre as 1000 portas mais comuns da sua própria máquina, onde o alvo (DVWA) está rodando na porta 8080.
Starting Nmap scan...
PORT STATE SERVICE
8080/tcp open http-proxy ← nosso alvo!
1.3
Detectar versões dos serviços (-sV)
KALI — root@kali
root@kali#nmap -sV -p 8080 localhost
-sV tenta descobrir o nome e a versão exata do serviço em cada porta.
-p 8080 limita a varredura apenas à porta 8080 — mais rápido e direto.
PORT STATE SERVICE VERSION
8080/tcp open http Apache httpd 2.4.x
← Apache encontrado! Versão revelada.
💡 POR QUE A VERSÃO IMPORTA?Com a versão exata do Apache, um atacante real buscaria no banco de dados CVE (cve.mitre.org) vulnerabilidades conhecidas para aquela versão específica.
1.4
Varredura com scripts padrão (-sC) e salvar resultado
KALI — root@kali
root@kali#nmap -sV -sC -p 8080 localhost -oN /tmp/scan-nmap.txt
-sC roda scripts automáticos do nmap: busca título da página, métodos HTTP permitidos, cookies e mais informações úteis.
-oN /tmp/scan-nmap.txt salva o resultado em arquivo — você vai anexar isso no relatório!
| http-title: Damn Vulnerable Web Application
| http-methods: GET POST HEAD OPTIONS
| http-auth: Basic realm="DVWA"

FASE 2 — VULNERABILIDADES WEB COM NIKTO
02

Escanear o site em busca de problemas conhecidos

O nikto testa automaticamente mais de 6700 vulnerabilidades e configurações erradas em servidores web.

niktoweb scanCVE
⏱ 15 min
🎯 O QUE VOCÊ VAI DESCOBRIR Arquivos perigosos expostos, configurações erradas, versões desatualizadas, cookies sem proteção e cabeçalhos HTTP de segurança ausentes.
2.1
Escanear o alvo com nikto
KALI — root@kali
root@kali#nikto -h http://localhost:8080 -o /tmp/scan-nikto.txt
-h http://localhost:8080 aponta o nikto para o nosso alvo.
-o /tmp/scan-nikto.txt salva todos os achados em arquivo para o relatório.
O scan leva entre 2 e 5 minutos. Cada linha com + é um achado!
+ Server: Apache/2.4.x — versão exposta no cabeçalho
+ /config/: Directory indexing found
+ Cookie PHPSESSID set without HttpOnly flag
+ No X-Frame-Options header found
+ No Content-Security-Policy header found
+ 6544 items checked: 23 error(s) and 8 item(s) reported
💡 LENDO OS RESULTADOSCada linha com + é um achado. Copie os mais importantes para o relatório. Achados com "OSVDB" ou "CVE" têm número de vulnerabilidade oficial — pesquise no Google para entender o impacto.
2.2
Ver o resultado salvo
KALI — root@kali
root@kali#cat /tmp/scan-nikto.txt
cat exibe o conteúdo do arquivo salvo. Este é o arquivo de evidências que você vai anexar ao relatório.
FASE 3 — INSPEÇÃO MANUAL COM CURL
03

Inspecionar cabeçalhos e resposta HTTP manualmente

O curl faz requisições HTTP e mostra exatamente o que o servidor responde — útil para confirmar achados do nikto.

curlHTTP headersfingerprint
⏱ 10 min
🎯 O QUE VOCÊ VAI DESCOBRIR Os cabeçalhos HTTP que o servidor envia — que revelam tecnologias, versões e configurações de segurança ausentes.
3.1
Ver cabeçalhos HTTP do servidor
KALI — root@kali
root@kali#curl -I http://localhost:8080
-I faz uma requisição HEAD — busca apenas os cabeçalhos, sem baixar o HTML.
Os cabeçalhos revelam: versão do servidor, linguagem usada, tipo de cookie e configurações de segurança.
HTTP/1.1 200 OK
Server: Apache/2.4.x ← versão exposta!
X-Powered-By: PHP/7.x ← linguagem exposta!
← SEM: Strict-Transport-Security
← SEM: X-Frame-Options
← SEM: Content-Security-Policy
⚠️ CABEÇALHOS AUSENTES = VULNERABILIDADETodo cabeçalho de segurança ausente é um achado real. Anote no relatório como criticidade Médio e explique por que cada um deveria estar presente.
3.2
Ver o HTML da página de login
KALI — root@kali
root@kali#curl http://localhost:8080/login.php | grep -i "form\|input\|action"
Baixa o HTML da página de login e filtra apenas as linhas com form, input e action — revela como o formulário funciona sem nem abrir o browser.
<form action="login.php" method="POST">
<input name="username" type="text">
<input name="password" type="password">
3.3
Testar login com credenciais padrão via curl
KALI — root@kali
root@kali#curl -X POST http://localhost:8080/login.php -d "username=admin&password=password&Login=Login" -L -v 2>&1 | grep "Location\|302\|200"
-X POST envia dados como formulário · -d define os campos · -L segue redirecionamentos · -v modo verboso.
As credenciais padrão do DVWA são admin / password. Um redirecionamento 302 indica login bem-sucedido!
HTTP/1.1 302 Found
Location: index.php ← login aceito com credencial padrão!
✅ ACHADO — CREDENCIAL PADRÃO ATIVAO sistema aceita admin/password — credenciais que nunca foram alteradas. Este é um achado de criticidade CRÍTICO para o relatório!
3.4
Salvar todas as evidências
KALI — root@kali
root@kali#curl -I http://localhost:8080 > /tmp/scan-curl.txt
Salva a saída dos cabeçalhos HTTP no arquivo de evidências.
root@kali#cat /tmp/scan-nmap.txt /tmp/scan-nikto.txt /tmp/scan-curl.txt > /tmp/evidencias-completas.txt
Junta todos os arquivos de saída em um único arquivo de evidências para anexar ao relatório.
